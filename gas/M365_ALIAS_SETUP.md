# Microsoft 365 メールエイリアス設定ガイド

GmailからM365のメールアドレス（@会社ドメイン）として送信・下書き作成するための設定手順。
この設定により、GASで作成した下書きの送信元がM365アドレスになる。

---

## 前提条件

- Microsoft 365 Business Premium（管理者権限あり）
- Gmail（Google Workspace）アカウント
- M365で多要素認証（MFA）が有効（Business Premiumでは通常有効）

---

## 事前準備（M365管理者として実施）

### 壁1: SMTP AUTH の有効化

Microsoftはセキュリティ強化のため、SMTP AUTH を**デフォルトで無効**にしている。
これをオンにしない限り、正しいパスワードを入力しても認証エラーになる。

1. [Microsoft 365 管理センター](https://admin.microsoft.com) を開く
2. **ユーザー** → **アクティブなユーザー** → 対象ユーザーを選択
3. **メール** タブ → **メール アプリの管理** をクリック
4. **「認証済み SMTP（Authenticated SMTP）」** にチェックを入れる
5. **保存**

> 反映まで最大30分かかる場合がある。

**PowerShellで設定する場合:**
```powershell
Connect-ExchangeOnline
Set-CASMailbox -Identity "you@yourcompany.com" -SmtpClientAuthenticationDisabled $false
```

### 壁2: セキュリティの既定値群（Security Defaults）

新しいM365テナントでは「セキュリティの既定値群」が有効になっている場合がある。
これが有効だと、**SMTP AUTH の個別許可が無視される**（全てのBasic認証がブロックされる）。

**確認方法:**
1. [Azure Portal](https://portal.azure.com) → **Microsoft Entra ID** → **プロパティ**
2. **セキュリティの既定値群の管理** を確認

**有効になっている場合の対処:**
- Business Premium には Azure AD P1 が含まれるため、**条件付きアクセスポリシー**に切り替え可能
- セキュリティの既定値群を無効化し、同等の条件付きアクセスポリシーで置き換える
- SMTP AUTH を使うユーザーだけ例外にするポリシーを作成

> ⚠️ セキュリティの既定値群の無効化はセキュリティレベルが下がるため、
> 必ず条件付きアクセスポリシーで代替すること。

### 壁3: アプリパスワードの発行（MFA有効時）

MFAが有効なアカウントでは通常のパスワードではSMTP認証できない。
専用のアプリパスワードを生成する。

1. [https://mysignins.microsoft.com/security-info](https://mysignins.microsoft.com/security-info) を開く
2. **「+ サインイン方法の追加」** → **「アプリ パスワード」** を選択
3. 名前を入力（例: `Gmail SMTP`）→ **作成**
4. 表示された16桁のパスワードをコピー（**これをGmailに入力する**）

> **管理者側の事前確認:** Entra ID → セキュリティ → 多要素認証 → 追加のクラウドベースMFA設定
> で「ユーザーがアプリ パスワードを作成できるようにする」が有効であること。

## Gmail側の設定

### エイリアス（Send mail as）の追加

1. PCブラウザで [Gmail](https://mail.google.com) を開く
2. **歯車アイコン** → **すべての設定を表示**
3. **「アカウントとインポート」** タブを開く
4. 「名前」セクションの **「他のメールアドレスを追加」** をクリック
5. ポップアップに情報を入力:

| 項目 | 値 |
|---|---|
| **名前** | 送信者として表示したい名前 |
| **メールアドレス** | M365のメールアドレス（例: `you@yourcompany.com`） |
| **エイリアスとして扱う** | 用途に応じてチェック（※下記参照） |

6. **次のステップ** をクリック
7. SMTPサーバー情報を入力:

| 項目 | 値 |
|---|---|
| **SMTPサーバー** | `smtp.office365.com` |
| **ポート** | `587` |
| **ユーザー名** | M365のメールアドレス（フルアドレス） |
| **パスワード** | 上記で発行した**アプリパスワード** |
| **暗号化** | **TLSを使用したセキュリティで保護された接続**（必須） |

> ⚠️ ポート465（SSL）は使用不可。必ず587（TLS/STARTTLS）を使うこと。

8. **アカウントを追加** をクリック

### 確認メールの承認

1. M365のメールボックス（Outlook/OWA）に確認メールが届く
2. メール内の**確認リンクをクリック**、または**確認コードをGmailのポップアップに入力**
3. 完了すると「送信者」として選択可能になる

### 「エイリアスとして扱う」の意味

- **チェックあり:** 返信がGmailの受信トレイに届く。GASでの処理に最適。
- **チェックなし:** 返信がM365のメールボックスに届く。

## トラブルシューティング

### 「認証に失敗しました」エラー

| 原因 | 確認ポイント | 対処 |
|---|---|---|
| SMTP AUTH 無効 | M365管理センター → メールアプリの管理 | 「認証済みSMTP」を有効化（反映まで最大30分待つ） |
| MFA + 通常パスワード | アプリパスワード未発行 | mysignins.microsoft.com でアプリパスワードを発行 |
| セキュリティの既定値群 | Azure Portal → Entra ID → プロパティ | 条件付きアクセスに切り替え |
| 条件付きアクセスポリシー | Entra ID → セキュリティ → 条件付きアクセス | レガシー認証ブロックポリシーにSMTP AUTH例外を追加 |

### デバッグ方法

**サインインログで原因を特定:**
1. Azure Portal → Microsoft Entra ID → **サインインログ**
2. ユーザーでフィルタ → クライアントアプリ「SMTP Auth」で絞り込み
3. エラーの詳細（エラーコード・理由）を確認

**コマンドラインでSMTP接続テスト:**
```bash
# swaks がインストールされている場合
swaks --to test@example.com --from you@yourcompany.com \
  --server smtp.office365.com --port 587 --tls \
  --auth-user you@yourcompany.com --auth-password "アプリパスワード"
```

## ⚠️ 将来的なリスク（重要）

Microsoftは段階的にSMTP AUTH のBasic認証を廃止する方針を示している。

| 時期 | 状況 |
|---|---|
| 2022年10月 | IMAP/POP のBasic認証廃止。**SMTP AUTH は猶予** |
| 2024年〜 | 新規テナントでアプリパスワード非対応のケースが増加 |
| 将来 | SMTP AUTH のBasic認証も完全廃止の可能性 |

**GmailのSend-As機能はOAuth2に対応していない**（パスワード認証のみ）ため、
MicrosoftがBasic認証を完全廃止した場合、この方法は使えなくなる。

**代替案（将来に備えて）:**
- Gmail転送 + 返信先ヘッダーでの運用（SMTP不要）
- M365側の「送信元アドレス」をGmailに変更
- GASから直接SMTP送信（OAuth2 XOAUTH2対応可能だが複雑）

**現時点では問題なく動作する。** ただし年に1回程度、Microsoftの方針変更を確認することを推奨。
